# 设置面板重设计方案

日期：2026-05-17

## 1. 目标

在**保留所有现有设置、按钮、下载区域、动态插槽**的前提下，重新组织设置面板里的元素布局，让它同时解决下面几个问题：

1. 设置太多，扫一眼时难以区分。
2. 当前 3 个选项卡的信息架构不稳定，同一类设置被拆散到了不同地方。
3. 下载进度区域和下载按钮被压在第二个选项卡里，开始下载后不够醒目。
4. 置顶功能虽然实用，但会进一步打散原有分类。
5. 当前面板更像“把所有设置排成一列”，效率高，但辨识度、层次感、可维护性都偏弱。

我的结论是：**最值得采用的方向，不是继续微调现在的 3 个横向选项卡，而是改成“左侧导航 + 右侧工作区”的结构。**

---

## 2. 已确认的约束

这次重设计不应该破坏下面这些既有能力：

- 保留所有现有设置项，不删功能。
- 保留现有按钮区、下载区、进度条区。
- 保留动态插槽，因为最终 HTML 是由多个模块拼出来的。
- 保留“置顶选项”机制。
- 保留“默认只显示少量常用项”的思路。
- 保留不同页面类型下的差异显示逻辑。
- 保留移动端适配能力。

换句话说，这次改的重点不是业务逻辑，而是**信息架构、布局层级、视觉组织方式**。

---

## 3. 我对现状的判断

### 3.1 目前的问题不只是“选项太多”

真正的问题其实有 4 个层面：

| 层面     | 当前情况                                     | 带来的问题                                         |
| -------- | -------------------------------------------- | -------------------------------------------------- |
| 一级结构 | 现在是 Crawl / Download / More 3 个选项卡    | More 承担了过多职责，像一个“杂物间”                |
| 二级结构 | More 里再分 5 个分类                         | 一部分同类设置又被拆到前两个选项卡，分类逻辑不稳定 |
| 行级结构 | 每个 option 基本都只是横向一行               | 当行内控件一多，就会变成一长串文字和开关           |
| 状态结构 | 置顶、高级设置、下载中、新功能等状态是叠加的 | 用户能操作，但不容易形成清晰心智模型               |

### 3.2 当前界面最核心的优点不能丢

虽然现在不够美观，但它有一个非常强的优点：**高频操作离结果很近**。

例如：

- 抓取相关按钮就在抓取设置附近。
- 下载相关设置和下载进度在同一个大区域。
- 大多数设置都能直接改，不需要经过太多层跳转。

所以重设计不能走成“更漂亮，但操作路径更长”。  
我的建议是：**保留效率优先，但把效率建立在更清晰的结构上，而不是建立在把所有东西直接摊开上。**

### 3.3 “显示高级设置”本质上是“简洁模式”

现在这个功能从体验上看，本质不是“显示高级设置”，而是：

- 默认进入简洁模式，只露出少量常用项。
- 用户需要时，再展开完整配置。

所以后续如果真的做界面改造，我建议把它从“某一个普通设置项”提升为**全局视图模式开关**。  
它应该出现在面板顶部工具区，而不是混在普通设置列表里。

---

## 4. 重设计时应该坚持的原则

### 4.1 先解决“找不到”，再解决“好不好看”

优先级应该是：

1. 让用户更快找到设置。
2. 让用户更容易理解某个设置属于哪一类。
3. 让下载状态更容易被看到。
4. 最后再做视觉美化。

### 4.2 一级分类要稳定，二级分组要贴近任务

一级分类不应该再出现“这类设置的一部分在 A，一部分在 B”的情况。  
一级分类一旦确定，就尽量稳定。

二级分组则应该更贴近用户任务，例如：

- “筛选抓取范围”
- “图片过滤”
- “命名规则”
- “保存格式”
- “下载行为”

而不是只按代码模块或历史演化痕迹分组。

---

## 5. 新的方案：标题栏 + 左侧导航 + 右侧工作区

这是我最推荐的方案。  
它比现在改动大一些，但收益也最大，而且最能同时兼顾：

- 结构清晰
- 高频操作效率
- 下载状态可见性
- 后续继续加设置时的可扩展性

### 5.1 总体框架

建议把整个面板改成下面这种结构：

```text
┌──────────────────────────────────────────────────────────────┐
│ 标题栏：logo、名称、搜索框、展开/折叠所有分组、GitHub、Wiki、赞助、关闭  │
├──────────────┬───────────────────────────────────────────────┤
│  主页        | 当前分类内容区                                  │
│  抓取        │ - 分组卡片                                     │
│  命名        │ - 每组可折叠                                   │
│  下载        │ - 每个设置为“标题 + 控件 + 子设置”结构           │
│  增强        │                                               │
│  其他        │                                               │
│  帮助        │                                               │
├──────────────│───────────────────────────────────────────────┤
│  下载摘要栏   │                                               │
└──────────────────────────────────────────────────────────────┘
```

标题栏的布局修改为左右两端分布：
- 左侧是 logo（文件是 `\src\static\icon\logo128.png`）和名字
- 右侧是搜索框等各种按钮、控件。

标题栏下面的结构里有几个关键变化：

1. **分类导航从顶部横向改成左侧纵向。**
2. 由于添加了左侧区域，所以设置面板需要适当加宽，这样右侧显示按钮、设置的区域依然可以使用与之前一致或接近的宽度，不会因为宽度变小而产生更多的换行。
3. **下载状态全局化**：在左侧导航区底部添加一个简单的显示下载状态摘要的区域，默认不显示内容，当下载开始后显示：下载总进度、暂停下载与停止下载按钮。后两个按钮不是必须的。

另外，移动端建议把中间部分（左侧导航区域+右侧分类内容区域）改成**上下布局**，而不是继续保留左右两列：

- 搜索框放在上方。
- 一级导航改成横向排列。
- 当前分类内容显示在最下方。

这样可以避免在窄屏上左右两列都过窄，导致大量换行。

### 5.2 一级导航建议改成 7 类

我建议把现在 More 里的 5 个大类真正提升为一级导航，这样结构会稳定很多。另外我还添加了 2 个额外的一级导航：“主页”和“帮助”。

| 一级导航 | 主要承载内容                                                                                            | 说明                                                                                 |
| -------- | ------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------ |
| 主页     | 置顶的设置、抓取按钮区域、其他按钮区域、下载区域                                                        | 该分类里的内容相当于仪表盘首页。该分类会默认显示                                     |
| 抓取     | 抓取范围、过滤条件、抓取策略、抓取按钮                                                                  | 对应“决定抓什么”                                                                     |
| 命名     | 文件名规则、文件夹规则、序号、标签别名                                                                  | 对应“决定文件叫什么”                                                                 |
| 下载     | 保存格式、下载线程、下载策略、下载触发                                                                  | 对应“决定如何保存和下载”                                                             |
| 增强     | 缩略图、预览、复制、高亮、界面增强                                                                      | 对应“增强使用过程”                                                                   |
| 其他     | 外观、语言、日志、导入导出、设置管理                                                                    | 系统级、全局级选项                                                                   |
| 帮助     | 显示原本位于下载器底部的帮助按钮和链接：常见问题 使用手册 GitHub 获取帮助 最近更新 Fanbox 下载器 赞助我 | 帮助信息和一些其他链接。它应该与上面的分类有视觉区分，因为它右侧显示的并不是设置项。 |

### 5.3 样式方面的一些设想

参考 Clash Verge 的配色方案（截图文件是 `\notes\settings panel\20260518_024810.png`）：
- 设置面板的左侧、右侧区域可以使用不同灰度的背景颜色来区分。
- 右侧的按钮区域、设置项都可以添加白色背景，并添加 padding、margin，使每个按钮区域或设置项都显示为独立的小卡片。这有助于设置项挨在一起时不容易区分开的问题：之前所有设置项都是直接显示文字，排列在一起时很难分辨不同设置的边界。做成小卡片之后就很容易分辨了。

附加说明：
1. 卡片要轻，不要做成太厚重的仪表盘风格。因为设置数量很多，如果每个设置卡片阴影很重、边框很深、圆角很大，界面会显得碎。更稳的是：
  - 浅色背景
  - 很轻的边框或阴影
  - 小圆角
  - 主要靠留白和分组来区分
  - 每个卡片之间有一定间距
2. 卡片分为这些种类：
  - 分类卡片（带标题）：用来显示分类标题，文字加粗加大显示。在右侧还有三角形箭头表示展开、折叠状态。点击一级卡片会切换其下二级卡片的展开/折叠状态。
  - 全宽度卡片：用于直接显示内容的卡片。没有标题、三角箭头等结构，只是单纯的卡片。根绝它是否隶属于分类卡片，其宽度会有区别：如果隶属于分类卡片，那么它的宽度比分类卡片少窄一些，以表示层级关系。如果它不属于分级卡片，那么就是全宽度。
  - 部分宽度的卡片：用于直接显示内容的卡片。没有标题、三角箭头等结构，只是单纯的卡片。它只占据部分宽度，通常用来显示小的按钮、超链接，所以不需要使用全宽度。不需要为它单独做一个卡片类型，因为只要为这些卡片的父元素设置 Grid 布局即可让卡片只占据部分宽度。
3. 右侧的卡片如何显示：
  - “首页”的右侧内容区域：有 4 个子区域，每种区域都是分类卡片+全宽度卡片的组合。
  - 中间 5 个设置分类的右侧内容区域：里面的内容是多个二级分类的设置，每种分类都是分类卡片+全宽度卡片的组合。
  - “帮助”分类：右侧的区域不是设置项，而是多个小按钮或超链接。所以它右侧没有分类卡片，只有多个占据了部分宽度的卡片。

### 5.4 置顶的设置如何处理

我在左侧导航区域添加了“首页”分类，它右侧的内容区域里会显示置顶的设置。

**被置顶的设置只有一份对应的设置元素：**

当左侧导航区域激活了“主页”分类时，查找所有被置顶的设置，把它们的元素从其原本所属的分类区域里移动到置顶分类的区域里。

当左侧导航区域激活了非“主页”分类时，则查找“主页”分类里是否有该分类下的设置，如果有则把这个设置从“主页”分类区域里移动到当前分类区域里。这样，用户在其他分类里依然能看到被置顶的设置，不会产生“这个设置怎么不见了”的困惑。

这样做的目的是为了让置顶的设置只有一份元素，而非多份。原因是下载器在设计时没有考虑到一个设置会存在多份的情况。如果一个设置存在多份，会带来问题：
- 由于设置项都包裹在一个 form 表单里，并且使用 form.key 的方式来读取、设置对应的值。如果被置顶的设置存在多份，那么同一个 key 会出现多次，造成问题。
- 一些模块需要向特定的设置里添加子元素，如果有多个相同的设置，就需要遍历多个设置来设置数据、创建子元素，会比较麻烦。

### 5.5 每个一级导航内部，再拆成“任务型分组”

下面是我建议的二级分组方式。

#### 抓取

| 分组           | 内容举例                                               |
| -------------- | ------------------------------------------------------ |
| 基础范围       | 抓取多少作品、抓取多少页面、抓取每个用户最新作品       |
| 作品类型       | 插画/漫画/动图/小说、年龄限制、AI、原创/非原创         |
| 收藏与关系过滤 | 收藏状态、收藏数、日均收藏、用户屏蔽、特定用户屏蔽 tag |
| 图像条件过滤   | 宽高、比例、尺寸、体积、图片数量、多图前后若干张       |
| 标签与标题过滤 | 需要 tag、不需要 tag、标题包含/排除、系列标题参与      |
| 执行策略       | 减慢抓取速度、定时抓取、自动导出结果、导出 ID          |
| 抓取操作       | 开始抓取、停止抓取、选中作品相关按钮                   |

#### 命名

| 分组         | 内容举例                                                |
| ------------ | ------------------------------------------------------- |
| 命名规则     | 图像命名规则、小说命名规则、按页面类型使用不同规则      |
| 文件夹规则   | 多图加文件夹、R18 文件夹、按 tag 建文件夹               |
| 特殊命名逻辑 | 标签别名、含特定 tag 用不同规则、用户名处理、去除 emoji |
| 序号与长度   | 序号起始值、补零、文件名长度限制、是否省略序号          |
| 命名辅助操作 | 保存命名规则、导入/选择规则                             |

#### 下载

| 分组           | 内容举例                                               |
| -------------- | ------------------------------------------------------ |
| 基础下载行为   | 下载线程、自动开始下载、下载间隔、保存到上次位置       |
| 文件格式       | 图片尺寸、动图保存格式、动图转换线程、小说保存格式     |
| 内容附加保存   | 保存作品元数据、保存作品简介、小说元数据、封面、内嵌图 |
| 下载策略       | 去重、下载顺序、点击收藏/点赞时下载、不下载已下载作品  |
| 系列与小说     | 自动合并系列小说、合并阈值、是否跳过单独下载           |
| 下载记录与状态 | 管理下载记录、导出结果、下载相关按钮                   |

#### 增强

| 分组         | 内容举例                                               |
| ------------ | ------------------------------------------------------ |
| 预览         | PreviewWork、预览尺寸、等待时间、单图/多图/动图预览    |
| 缩略图与列表 | 更大缩略图、缩略图按钮、边框、列表显示、多图缩略图列表 |
| 复制与信息   | 复制按钮、复制格式、详细信息面板                       |
| 页面增强     | 高亮关注用户、搜索结果过滤、移除已关注用户作品等       |

#### 其他

| 分组       | 内容举例                                 |
| ---------- | ---------------------------------------- |
| 外观       | 主题、背景图、关键字高亮、选项卡切换方式 |
| 语言与提示 | 语言、帮助提示重新显示                   |
| 日志       | 默认可见性、导出日志、导出时机、日志过滤 |
| 设置管理   | 导入设置、导出设置、重置设置             |

这个分法的重点是：**二级分组不是按历史文件结构来分，而是按用户决策流程来分。**

### 5.6 单个设置项应该改成“卡片化行结构”

现在很多设置都是“一行内堆很多控件”，看起来像连续文本。  
建议把每个设置统一成下面这种结构：

```text
┌─────────────────────────────────────────────────────┐
│ 标题区：设置名 | 帮助 | wiki | 状态徽标（NEW / 已置顶） │
│ 主控件区：开关 / 单选 / 输入框 / 组合控件             │
│ 子设置区：仅在启用后展开，缩进显示                     │
└─────────────────────────────────────────────────────┘
```

具体建议：

- 每个设置块之间增加更稳定的垂直间距。
- 标题与控件分成两层，而不是永远并排。
- 有子设置的 option，子区域用浅背景、缩进、细边框区分。
- 帮助按钮和 Wiki 链接收拢到标题右侧，减少正文噪音。
- `NEW`、`📌` 之类状态不要再插在左侧悬空位置，改成标题区徽标更稳。

这样做以后，哪怕选项很多，扫描负担也会明显下降。

### 5.8 抓取按钮、下载按钮、其他功能按钮的摆放方式

按钮区不建议再简单放在设置流里，和普通设置项混排。  
但**抓取相关按钮也不应该只放进“抓取”分类页里**，否则用户必须先切换分类才能点击“开始抓取”，这会伤害高频操作效率。

目前的方案是放在“首页”分类（仪表盘）里。

### 5.9 “显示高级设置”开关改为“展开/折叠所有分组”

因为现在每个一级导航（分类）里对设置项进行了二级分组，每个分组里的设置数量不太多，因此没必要默认隐藏部分设置了。

同时，由于增加了二级分组，并且是可以展开/折叠的，所以把“显示高级设置”开关改为“展开/折叠所有分组”按钮，放在标题栏右侧区域。

### 5.10 这个方案为什么最适合这个项目

因为这个项目的本质不是一个普通设置页，而是一个：

- 带运行状态的工具面板
- 带大量专业设置
- 同时要求高频操作效率很高

所以它更像：

- **轻量控制台**
- 而不是传统的“系统设置页”

用“左侧导航 + 右侧工作区”的方式，会比传统选项卡更适合这种工具属性。

## 7. 我最建议先落地的 5 个改动

如果不想一次改完全部，可以按收益排序分阶段做。


### 第 3 步：把 option 改成卡片化行结构

哪怕一级导航暂时不动，只做这一项，也能明显改善辨识度。

### 第 4 步：加入搜索框和筛选器

对于近百个设置，搜索几乎一定值得做。

### 第 5 步：把 5 个分类提升为一级导航

这一步是结构终局，能彻底解决 More 过重的问题。

## 8. 如果以后真的开始实现，我建议的代码层面拆法

虽然这份文档现在只是设计思路，但如果后续要落地，我建议从代码结构上这样拆：

### 8.1 先补一份“设置元数据表”

不要继续只靠 `data-no` 和写死顺序来表达结构。  
建议给每个设置补充元数据，例如：

- 所属一级导航
- 所属二级分组
- 是否被置顶。默认都是 `false`，用户手动把它置顶之后为 `true`。
- 搜索关键词

这样后面无论做：

- 分组折叠
- 搜索设置
- 置顶设置
- 当前页面过滤

都会容易很多。

### 8.2 HTML 层面要从“连续列表”改成“分区容器”

当前结构更像是一个连续流。  
后续应该变成：

- nav 容器
- toolbar 容器
- quick access 容器
- section group 容器

### 8.3 置顶逻辑建议从“移动到选项卡顶部”改成“移动到首页里”

这样可以尽量少碰原有分类顺序，也更符合用户心智。

---

## 9. 最终结论

如果只用一句话概括我的建议，就是：

> 把当前设置面板从“3 个页签里的长表单”，升级成“带仪表盘（首页）、导航分类、全局下载状态”的轻量控制台。

其中最关键的 4 个点是：

1. **一级分类稳定化**：把 5 个分类真正提升为主导航。
2. **在仪表盘里集中显示置顶的设置和多种按钮**：把这些重要的设置和功能集合起来，而不是分散在多个选项卡里。
3. **下载状态全局化**：在左侧导航区底部添加一个简单的显示下载状态摘要的区域，默认不显示内容，当下载开始后显示：下载总进度、暂停下载与停止下载按钮。。
4. **设置项卡片化**：每个 option 都有更清晰的标题、控件、子设置层次。

如果你后续要我继续推进，我建议下一步不是直接写最终代码，而是先做两件事：

1. 先把所有设置项映射到“一级导航 + 二级分组 + 常用/高级”的结构表。
2. 再基于这张结构表，画一版更接近真实界面的低保真原型。
